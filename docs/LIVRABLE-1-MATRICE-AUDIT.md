## Matrice d'Audit — Pratiques A à E vs Règles ANSSI

### Pratique A — Topologie : Réseau plat unique (10.0.0.0/8)

| Élément                  | Détail                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description**          | Tout le réseau (serveurs Web, bases de données MySQL, PC comptabilité, Wi-Fi invité) connecté sur un seul commutateur géant avec l'adressage 10.0.0.0/8                                                                                                                                                                                                 |
| **Règles ANSSI violées** | **Règle 19** — Segmenter le réseau et mettre en place un cloisonnement entre les zones réseau                                                                                                                                                                                                                                                           |
|                          | **Règle 22** — Interdire la connexion directe à Internet des postes ou serveurs sensibles                                                                                                                                                                                                                                                               |
|                          | **Règle 15** — Protéger les informations sensibles — le mélange Wi-Fi invité + BDD bancaires est une faute grave                                                                                                                                                                                                                                        |
| **PCI-DSS**              | **Exigence 1.2** — Restreindre les connexions entre les réseaux non approuvés et les composants du CDE                                                                                                                                                                                                                                                  |
|                          | **Exigence 1.3** — Interdire l'accès direct entre Internet et le CDE                                                                                                                                                                                                                                                                                    |
| **Risque cyber**         | **Mouvement latéral total** : un attaquant ayant compromis un seul poste (ex: Wi-Fi invité ou PC comptabilité) accède directement aux serveurs MySQL contenant les données bancaires. Aucune barrière entre les zones de confiance (DMZ, LAN, invités, production). C'est exactement ce qui a permis le déploiement du ransomware sur l'ensemble du SI. |
| **Criticité**            | 🔴 **CRITIQUE**                                                                                                                                                                                                                                                                                                                                         |
| **Remédiation Azure**    | Architecture **Hub & Spoke** avec VNets séparés par zone de confiance. Micro-segmentation via **NSG + ASG** (Application Security Groups) par tier applicatif. Modèle **Zero Trust** : tout trafic inter-tiers est refusé par défaut (deny-all) et autorisé explicitement. Isolation CDE (données bancaires) dans un VNet dédié (Spoke-Data).           |

### Pratique B — Administration : IP publiques sur les BDD + SSH root + mot de passe faible

| Élément                  | Détail                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description**          | Serveurs MySQL avec adresses IP publiques directes. Administration SSH (port 22) depuis Internet avec le compte `root` et le mot de passe `P@ssword123`                                                                                                                                                                                                                                                                               |
| **Règles ANSSI violées** | **Règle 9** — Ne pas donner les droits administrateur à un utilisateur n'en ayant pas le besoin (utilisation du compte root pour l'administration courante)                                                                                                                                                                                                                                                                           |
|                          | **Règle 10** — Définir et vérifier des règles de choix et de dimensionnement des mots de passe (mot de passe trivial, aucune politique de complexité)                                                                                                                                                                                                                                                                                 |
|                          | **Règle 14** — Mettre en place des mécanismes d'authentification forte (mot de passe seul, pas de MFA, pas de clés SSH)                                                                                                                                                                                                                                                                                                               |
|                          | **Règle 22** — Interdire la connexion directe à Internet des serveurs de production (IP publiques sur les BDD)                                                                                                                                                                                                                                                                                                                        |
|                          | **Règle 28** — Protéger les flux d'administration — toute connexion d'administration transitant par un réseau non maîtrisé doit utiliser un protocole chiffré et authentifié dans un canal dédié (administration via Internet sans tunnel chiffré dédié = violation directe)                                                                                                                                                         |
| **PCI-DSS**              | **Exigence 2.1** — Ne pas utiliser les valeurs par défaut du fournisseur pour les mots de passe système                                                                                                                                                                                                                                                                                                                               |
|                          | **Exigence 8.2** — Attribuer un ID unique à chaque personne ayant un accès informatique                                                                                                                                                                                                                                                                                                                                               |
|                          | **Exigence 8.5** — Ne pas utiliser de comptes génériques partagés                                                                                                                                                                                                                                                                                                                                                                     |
| **Risque cyber**         | **Compromission immédiate** : les serveurs de BDD sont exposés directement sur Internet avec un mot de passe trivial sur le compte root. N'importe quel scanner automatisé (Shodan, Censys) peut découvrir ces serveurs et forcer l'accès en quelques secondes. Le compte root donne un contrôle total sur la base de données ET le système d'exploitation. Facteur aggravant : aucune traçabilité d'accès (pas de compte nominatif). |
| **Criticité**            | 🔴 **CRITIQUE**                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Remédiation Azure**    | **Aucune IP publique** sur les VMs. Administration exclusivement via **Azure Bastion** (tunnel SSH/RDP natif via le portail Azure, sans exposition de port). Authentification par **clés SSH Ed25519** (pas de mots de passe). Comptes **nominatifs** (pas de root direct). Les BDD sont dans un subnet isolé sans route vers Internet.                                                                                               |

### Pratique C — Hybridation : Tunnel GRE en clair sur Internet

| Élément                  | Détail                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description**          | L'agence de Lyon est reliée au siège par un simple tunnel GRE en clair traversant Internet                                                                                                                                                                                                                                                                                                         |
| **Règles ANSSI violées** | **Règle 25** — Sécuriser les interconnexions réseau dédiées avec des tiers (aucun chiffrement sur le lien inter-sites)                                                                                                                                                                                                                                                                             |
|                          | **Règle 19** — Segmenter le réseau (le tunnel GRE crée un pont direct sans filtrage entre les deux sites)                                                                                                                                                                                                                                                                                          |
|                          | **Règle 29** — Chiffrer les données sensibles transmises par voie réseau (données bancaires transitant en clair)                                                                                                                                                                                                                                                                                   |
| **PCI-DSS**              | **Exigence 4.1** — Utiliser des protocoles de chiffrement et de sécurité forts pour protéger les données sensibles lors de la transmission sur les réseaux ouverts                                                                                                                                                                                                                                 |
| **Risque cyber**         | **Interception totale du trafic** : le protocole GRE (RFC 2784) est un tunnel d'encapsulation sans AUCUN mécanisme de chiffrement, d'intégrité ni d'authentification. Tout acteur ayant accès au chemin réseau (ISP, attaquant MITM) peut capturer le trafic en clair. Les données bancaires, identifiants d'administration et flux internes sont lisibles par n'importe quel sniffer (Wireshark). |
| **Criticité**            | 🔴 **CRITIQUE**                                                                                                                                                                                                                                                                                                                                                                                    |
| **Remédiation Azure**    | Remplacement par un tunnel **IPsec IKEv2** avec **BGP** via Azure VPN Gateway. Chiffrement AES-256-GCM, intégrité SHA-384, authentification par Pre-Shared Key forte (> 32 caractères). Échange de routes dynamique via BGP (AS privés distincts). Perfect Forward Secrecy (PFS) activé.                                                                                                           |

### Pratique D — Traçabilité : Aucune journalisation centralisée

| Élément                  | Détail                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description**          | Aucun serveur central de journalisation. Les logs des routeurs sont uniquement en RAM et disparaissent à chaque redémarrage                                                                                                                                                                                                                                                                                                                                          |
| **Règles ANSSI violées** | **Règle 36** — Activer et configurer les journaux des composants les plus importants (aucun journal persistant)                                                                                                                                                                                                                                                                                                                                                      |
|                          | **Règle 37** — Définir et appliquer une politique de journalisation (aucune politique, aucune rétention)                                                                                                                                                                                                                                                                                                                                                             |
|                          | **Règle 38** — Procéder à des contrôles et audits de sécurité réguliers (impossible sans logs)                                                                                                                                                                                                                                                                                                                                                                       |
|                          | **Règle 40** — Définir une procédure de gestion des incidents de sécurité (pas de données forensiques exploitables)                                                                                                                                                                                                                                                                                                                                                  |
| **PCI-DSS**              | **Exigence 10.1** — Implémenter des pistes d'audit pour relier tous les accès aux composants du système à chaque utilisateur individuel                                                                                                                                                                                                                                                                                                                              |
|                          | **Exigence 10.5** — Sécuriser les pistes d'audit de manière à ce qu'elles ne puissent être modifiées                                                                                                                                                                                                                                                                                                                                                                 |
|                          | **Exigence 10.7** — Conserver l'historique des pistes d'audit pendant au moins un an                                                                                                                                                                                                                                                                                                                                                                                 |
| **Risque cyber**         | **Aveuglement total** : sans logs centralisés, l'entreprise est incapable de détecter une intrusion en cours, d'analyser un incident post-mortem ou de prouver la conformité réglementaire. Les logs en RAM disparaissent au redémarrage — or c'est exactement ce qu'un attaquant fait après compromission (reboot pour effacer ses traces). L'enquête forensique post-ransomware est impossible, la Banque de France ne peut obtenir aucune preuve d'investigation. |
| **Criticité**            | 🔴 **CRITIQUE**                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Remédiation Azure**    | Déploiement d'un **Log Analytics Workspace** (SIEM) centralisé. **NSG Flow Logs v2** avec Traffic Analytics. **Azure Monitor Agent (AMA)** sur toutes les VMs avec Data Collection Rules (Syslog + Performance). **Diagnostic Settings** sur le Firewall, le Bastion et les VPN Gateways. Storage Account dédié pour archivage long terme (30 jours minimum, 1 an recommandé). **Action Groups** pour alerting automatique (email SecOps).                           |

### Pratique E — Filtrage Sortant : Accès Internet total depuis la production

| Élément                  | Détail                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Description**          | Les serveurs de production peuvent naviguer librement sur Internet (HTTP/HTTPS ouvert en 0.0.0.0/0 sortant). Le ransomware a pu communiquer avec son serveur C2 (Command & Control)                                                                                                                                                                                                                                                                                        |
| **Règles ANSSI violées** | **Règle 22** — Interdire la connexion directe à Internet des postes ou serveurs sensibles                                                                                                                                                                                                                                                                                                                                                                                  |
|                          | **Règle 23** — Utiliser un proxy et des passerelles de sécurité pour l'accès Internet                                                                                                                                                                                                                                                                                                                                                                                      |
|                          | **Règle 24** — Contrôler et protéger l'accès aux salons et aux forums de discussion, et aux sites non liés à l'activité professionnelle                                                                                                                                                                                                                                                                                                                                    |
|                          | **Règle 19** — Segmenter le réseau (absence de DMZ, pas de zone de filtrage)                                                                                                                                                                                                                                                                                                                                                                                               |
| **PCI-DSS**              | **Exigence 1.2.1** — Restreindre le trafic entrant et sortant au strict nécessaire pour l'environnement de données des titulaires de carte                                                                                                                                                                                                                                                                                                                                 |
|                          | **Exigence 1.3.4** — Ne pas autoriser le trafic sortant non autorisé depuis le CDE vers Internet                                                                                                                                                                                                                                                                                                                                                                           |
| **Risque cyber**         | **Communication C2 et exfiltration** : c'est la faille qui a directement permis au ransomware de fonctionner. Sans filtrage sortant, un malware peut : (1) Télécharger ses modules depuis Internet, (2) Communiquer avec son serveur Command & Control (C2) pour recevoir des instructions, (3) Exfiltrer des données bancaires vers des serveurs attaquants, (4) Propager l'infection via des connexions sortantes. Le 0.0.0.0/0 ouvert est l'absence totale de contrôle. |
| **Criticité**            | 🔴 **CRITIQUE**                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Remédiation Azure**    | **Forced Tunneling** via UDR (User Defined Routes) : tout trafic sortant (0.0.0.0/0) est redirigé vers **Azure Firewall** centralisé dans le Hub. Le Firewall applique des règles de filtrage applicatives (FQDN) — seules les destinations légitimes sont autorisées (mises à jour Ubuntu, Azure Monitor, etc.). Tout autre trafic sortant est bloqué (deny by default). Threat Intelligence activé sur le Firewall pour bloquer les domaines/IP malveillants connus.     |

## Synthèse des Écarts

| Pratique               | Règles ANSSI           | PCI-DSS          | Criticité   | Remédiation principale                |
| ---------------------- | ---------------------- | ---------------- | ----------- | ------------------------------------- |
| **A** — Réseau plat    | R19, R22, R15          | 1.2, 1.3         | 🔴 CRITIQUE | Hub & Spoke + NSG/ASG Zero Trust      |
| **B** — Admin publique | R9, R10, R14, R22, R28 | 2.1, 8.2, 8.5    | 🔴 CRITIQUE | Bastion + clés SSH + zéro IP publique |
| **C** — GRE en clair   | R25, R19, R29          | 4.1              | 🔴 CRITIQUE | IPsec IKEv2 + BGP                     |
| **D** — Pas de logs    | R36, R37, R38, R40     | 10.1, 10.5, 10.7 | 🔴 CRITIQUE | Log Analytics + AMA + Flow Logs       |
| **E** — Egress ouvert  | R22, R23, R24, R19     | 1.2.1, 1.3.4     | 🔴 CRITIQUE | Forced tunneling → Azure Firewall     |

**Conclusion audit** : Les 5 pratiques identifiées sont **toutes critiques**. Elles couvrent l'ensemble des piliers de la sécurité réseau (segmentation, authentification, chiffrement, traçabilité, filtrage). L'infrastructure existante ne respecte aucune des exigences fondamentales du Guide d'hygiène ANSSI ni de PCI-DSS. La migration vers Azure avec une architecture "Secure by Design" est impérative.
